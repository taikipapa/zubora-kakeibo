# ズボラ家計簿 設計図

## 1. アプリ概要

「ズボラ家計簿」は、財布ごとの残高を直感的に管理するスマホアプリ（React Native + Expo）。家計簿的な分析機能ではなく、「財布の中身がどう増減したか」を楽しく見える化することに特化する。

## 2. コンセプト

- カテゴリ分けなしの「超シンプル残高管理」
- 操作は「財布を選ぶ → 入る/出るを選ぶ → 金額を入力 → 保存」の4ステップのみ
- ゲーム的な財布アニメーションで操作の楽しさを担保する

## 3. このアプリでやること

- 財布の作成（広告視聴後、最大5個）
- 入る/出るの記録と残高の即時反映
- 履歴の表示・削除（削除時は残高を復元）
- 財布アニメーション（ぷにっと膨らむ/しぼむ）
- テーマ切り替え（わいわい / ほっこり）

## 4. このアプリでやらないこと

- カテゴリ分類（食費・交通費など）
- メモ・店名・支払い方法・レシート・タグ入力
- 予算管理・グラフ分析・月次集計
- 財布名の自動命名
- 複数財布の同時表示（1画面1財布のみ）

## 5. 財布仕様

| 項目 | 内容 |
|---|---|
| 初期財布数 | 1個 |
| 初期財布名 | ズボラ財布 |
| 初期財布タイプ | がま口財布 |
| 初期残高 | 0円 |
| 最大財布数 | 5個 |
| 表示方法 | 1画面1財布、大きく表示 |
| 切り替え方法 | 左右スワイプ／ページ切り替え |
| 財布番号表示 | 「1 / 5」のように表示 |
| 削除制約 | 最後の1個は削除不可 |
| 削除時の挙動 | その財布の履歴も全削除 |

## 6. 財布追加仕様

1. 「財布を追加」ボタンを押す
2. 「広告を見ると財布を1つ追加できます」と表示
3. ユーザーがリワード広告を視聴
4. 広告視聴完了後、財布作成モーダルを表示
5. ユーザーが財布名を入力（必須・空欄不可）
6. ユーザーが財布タイプを選択（4種から1つ、デフォルト選択でも可）
7. 「作成」で財布を追加し、新規財布のページへ自動遷移

バリデーション：
- 財布名が空欄 → 作成ボタンを非活性
- 財布が既に5個 → 「財布を追加」ボタン自体を非活性 or 上限メッセージ表示

## 7. 広告仕様

- 広告の種類：リワード広告（視聴必須、スキップ不可が前提）
- 広告視聴は「財布追加」のトリガーのみに使用（他機能には使わない）
- 広告失敗・キャンセル時：財布作成モーダルを表示しない（追加処理を中断）
- 広告SDKは未確定（AdMob想定）。MVPでは広告呼び出し部分をインターフェース化し、後でSDK実装を差し替えられるようにする（`RewardAdService`のような抽象を置く）
- 設定画面に「広告について」の説明枠を用意（広告ポリシー等の簡易テキスト）

## 8. 入出金仕様

入力項目：
- 入る / 出る（トグルまたは2ボタン）
- 金額（数値入力）

対象財布：現在表示中の財布（明示選択UIなし）

日時：自動で現在日時を記録（ユーザー入力なし）

入る場合：
- `wallet.balance += amount`
- 履歴追加：`{ walletName, type: 'income', amount, createdAt }`

出る場合：
- `wallet.balance -= amount`（残高が負になることを許容するか要検討。MVPでは許容＝制限なしを推奨。ズボラ家計簿のコンセプト上、厳密なエラー処理は不要）
- 履歴追加：`{ walletName, type: 'expense', amount, createdAt }`

バリデーション：
- 金額が0または未入力 → 保存不可
- 金額は正の数値のみ

## 9. アニメーション仕様

**MVP実装（簡易版）：**

| 操作 | アニメーション |
|---|---|
| 入る | 財布を少し拡大（scale up）＋軽いバウンド |
| 出る | 財布を少し縮小（scale down）＋軽い揺れ（shake） |
| 共通 | 残高数字を即時更新（カウントアップ/ダウンも検討可だがMVPでは即時反映でOK） |

**将来拡張を見据えた構成：**
- アニメーションロジックを `useWalletAnimation` のようなフック／モジュールに分離
- 「お金アイコンが下から財布に向かって移動する」演出を後で追加できるように、アニメーション発火のタイミング（保存ボタン押下時）とコンポーネント構造（入力エリア・財布表示エリアを別コンポーネントとして分離）を最初から意識する
- アニメーション用の値（scale, rotateなど）はAnimated Valueとして集中管理し、新しいエフェクト（移動するコインアイコンなど）を追加で重ねやすい設計にする

## 10. テーマ仕様

2テーマを1アプリ内で切り替え可能。初期は「わいわいテーマ」。設定画面で「ほっこりテーマ」に変更可。選択状態はローカル保存（AsyncStorage想定、理由は18章）。

2テーマとも、UI構造（ヘッダー／財布表示／入出金ボタン／金額入力／保存・キャンセル／広告バナー／履歴リスト）は共通。差分は**配色・装飾・コンポーネントの形状・残高表示の見せ方**にある。

> 注：参考画像内の履歴文言「コンビニで前払い」「ランチ代」「おにぎり購入」はデザイン例。実際のMVPでは履歴に表示するのは「財布名／入った・出た／金額／日時」のみで、メモ・内容文言は表示しない（ほっこり画像の履歴行「ズボラ財布 +1,000円 今日10:15」がMVPの実際の表示形式に近い）。

## 11. 履歴仕様

表示項目（これ以外は表示しない）：
- 日時
- 財布名
- 入った / 出た
- 金額

履歴は全財布共通の一覧として表示（財布ごとの絞り込みは任意機能としてMVP後検討）。
新しい順に表示。
各履歴行に削除アクション（スワイプ削除 or 削除ボタン）を用意。

## 12. 履歴削除時の残高復元ルール

履歴削除時、対応する取引を打ち消す形で財布残高を復元する。

- 削除対象が `income`（入った）の履歴 → `wallet.balance -= amount`
- 削除対象が `expense`（出た）の履歴 → `wallet.balance += amount`

その後、該当の履歴レコードを削除する。
財布が既に削除されている場合（整合性の都合）は復元処理をスキップし、履歴のみ削除（通常は13章のルールにより財布削除時に履歴も消えるため、このケースは基本発生しない）。

## 13. 財布削除時の履歴削除ルール

財布を削除する際：
1. 対象 `walletId` に紐づく `Transaction` を全件削除
2. 対象 `Wallet` を削除
3. 削除後、表示中ページを他の財布（例：先頭の財布）に切り替え、ページ番号表示を更新

制約：財布が1個のみの場合は削除不可（5章参照）。削除確認ダイアログを表示すること（取り消せない操作のため）。

## 18. 保存方式の提案

**比較：AsyncStorage vs SQLite**

| 観点 | AsyncStorage | SQLite (expo-sqlite) |
|---|---|---|
| データ構造 | キーバリュー、JSON文字列で保存 | リレーショナル、テーブル定義可能 |
| 整合性 | 自前でJSON全体読み書き、複数操作の同時実行で不整合リスクあり | トランザクション機能でwallet残高更新とtransaction挿入をアトミックに実行可能 |
| 履歴の増加耐性 | 件数増加でJSON全体read/writeが重くなる | インデックス・クエリで件数増加に強い |
| 削除/復元処理 | 配列operationを自前実装 | SQLトランザクションで「履歴削除＋残高復元」を1単位として安全に実行 |
| 実装コスト | 低い | やや高い（スキーマ・マイグレーション管理が必要） |

**結論：SQLite（`expo-sqlite`）を推奨**

理由：
- 「入出金で残高を更新」「履歴削除で残高を復元」という処理は、残高(Wallet)と履歴(Transaction)を同時に整合させる必要があり、SQLiteのトランザクションで原子性を保証できる
- 履歴は時間経過で増え続けるため、AsyncStorageのJSON全体read/write方式はパフォーマンス劣化のリスクがある
- 財布数は最大5個と少ないが、履歴は無制限に増える想定のため、テーブル設計の恩恵が大きい

**Settings（テーマ設定）のみAsyncStorageを使用**：単一の小さな値であり、SQLiteを使う必要がないため。`WalletとTransactionはSQLite、Settingsのみ AsyncStorage`という併用構成を提案。

## 19. アニメーション方式の提案

**比較：React Native Animated vs Reanimated**

| 観点 | Animated (標準API) | Reanimated (react-native-reanimated) |
|---|---|---|
| 実行スレッド | 基本JSスレッド（一部Native Driver対応） | UIスレッドで実行、JSスレッドのブロックに影響されにくい |
| スワイプ/ジェスチャー連携 | gesture-handlerとの連携がやや複雑 | gesture-handlerとの統合が標準的でスムーズ |
| 表現力 | scale/opacity等の基本アニメーションは可能 | より複雑な物理ベース（spring, bounce）アニメーションが書きやすい |
| Expo対応 | 標準搭載 | Expo SDKで公式サポート、設定も簡単 |
| 将来拡張（コイン移動等） | 可能だが複雑になりやすい | 複数要素の連動アニメーションを宣言的に書きやすい |

**結論：Reanimated を推奨**

理由：
- 財布の「ぷにっと膨らむ」「しぼんで揺れる」といったspring/bounce系の演出はReanimatedの`withSpring`/`withSequence`で自然に書ける
- 財布の左右スワイプ切り替えは`react-native-gesture-handler` + `Reanimated`の組み合わせ、または`react-native-pager-view`と組み合わせるのが一般的で相性が良い
- 将来「お金アイコンが移動するアニメーション」を追加する際も、共有値（shared values）でコイン位置・財布の状態を連動させやすい
- Expoとの統合が容易（Expo SDK標準サポート）

```
   │  │  [ホーム画面に戻る・新財布ページ表示]
   │  │
   │  ├─ 入出金保存 → 同画面内でアニメーション再生・残高更新
   │  │
   │  ├─ 履歴アイコンタップ
   │  │      ▼
   │  │   [履歴画面]
   │  │      ├─ 履歴削除（確認ダイアログ）→ 残高復元 → ホームに反映
   │  │      └─ 戻る → [ホーム画面]
   │  │
   │  └─ 設定アイコンタップ
   │         ▼
   │      [設定画面]
   │         ├─ テーマ切替 → 即時反映
   │         ├─ 全データリセット（確認ダイアログ）
   │         ├─ 広告について（説明表示）
   │         ├─ アプリ説明（説明表示）
   │         └─ 戻る → [ホーム画面]
   │
   └─ 財布削除（確認ダイアログ、ホーム画面内アクション）→ 履歴連動削除 → 他財布ページへ
```

## 24. 開発ステップ

1. プロジェクト初期化（Expo + TypeScript構成、必要ライブラリ選定確定）
2. SQLiteスキーマ定義・DBクライアント実装（Wallet/Transaction）
3. 初期財布のシード処理（初回起動時に「ズボラ財布」を作成）
4. ホーム画面：財布1個の静的表示（アニメーション無し）
5. 入出金機能：金額入力→保存→残高更新（DBトランザクション込み）の動作確認
6. 財布ページャー実装（スワイプ切り替え、ページ番号表示）
7. 財布追加フロー（広告ダミー→モーダル→作成、上限5個チェック）
8. 財布削除（履歴連動削除、最後の1個は不可）
9. 履歴画面（一覧表示、削除＋残高復元）
10. アニメーション実装（Reanimated導入、拡大バウンド／縮小シェイク）
11. テーマ機構実装（ThemeContext、わいわい/ほっこりトークン、AsyncStorage保存）
12. 設定画面（テーマ切替・全データリセット・広告について・アプリ説明）
13. 広告SDK本実装への差し替え（RewardAdServiceの実装部分）
14. 全体UI調整・装飾要素（コイン/キラキラ等のテーマ装飾）追加
15. 動作確認・最終調整

## 25. Claude Code が守る開発ルール

- カテゴリ・メモ・店名・支払い方法・レシート・タグ・予算分類に関する項目は、フィールド・UI・型定義のいずれにも追加しない
- 履歴に表示・保存する情報は「日時・財布名・入った/出た・金額」のみとする
- 財布残高の更新（入出金・履歴削除による復元）は、対応するDB操作と必ずセットで行い、不整合な中間状態を作らない（SQLiteトランザクションを使う）
- 財布は常に最大5個、最小1個の制約を外れないようにコード上でガードする（UIだけでなくロジック層でも検証する）
- 財布追加は必ず「広告視聴完了」を経てからモーダルを表示する。広告ステップをスキップする実装をしない
- テーマの色・装飾はテーマトークン経由でのみ参照し、コンポーネント内に色のハードコードをしない
- アニメーションはMVPでは拡大/縮小+簡易モーションのみとし、将来のコイン移動アニメーション追加を阻害しない形（アニメーション値・トリガーを分離した構成）を保つ
- 実装時は本設計図の範囲を超える機能（分析・グラフ・通知・複数言語など）を先回りして追加しない
- 不明点や設計の解釈が割れる箇所は、実装を進める前にユーザーに確認する
